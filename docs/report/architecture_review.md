# アーキテクチャ構成の見直しとリファクタリング報告書（2026-03-21）

## 1. 実施済みのリファクタリング概要

今回の作業では、プロジェクトのメンテナンス性と拡張性を高めるため、以下の**「関心の分離（Separation of Concerns）」**を徹底しました。

### 構成の三層分離
- **Route層 (`src/routes/*.schema.ts`)**: 
    - OpenAPI のスキーマ定義、Zod バリデーション、エンドポイントのメタデータのみを担当。
    - 実装（ロジック）を一切持たせないことで、API 仕様をひと目で把握可能に。
- **Handler層 (`src/handlers/*.handler.ts`)**: 
    - HTTP リクエストの受け取り、パラメータのパース、サービス（ロジック）の呼び出し、レスポンスの整形を担当。
    - `AppRouteHandler` 型を導入し、`Context` を介した bindings/variables へのアクセスを型安全に。
- **Service/Logic層 (`src/services/`, `src/repositories/`)**:
    - 純粋なビジネスロジックやデータベース操作を担当。
    - Hono の `Context` に依存しない設計にすることで、単体テストが容易な状態に。

### 基盤機能の共通化
- **Repository Injection**: ミドルウェア (`injectRepos`) により各リクスの `env` からリポジトリを自動生成し `c.get('repos')` で提供。
- **Global Error Handling**: 全体のエラー処理を `error.handler.ts` に集約。
- **JSDocの追加**: 各ハンドラーの役割（特にOAuthフローにおけるセキュリティ上の意図など）を明文化。

## 2. 現在の「ロジック分離」に関する分析

現在の実装全体を見直した結果、まだ **Handler 層に「ロジック（具体的な手順）」が漏れ出している（Leakageしている）箇所** がいくつか見受けられます。

### A. 認証フロー（Auth Handler）
現状、`googleCallbackHandler` が「APIへのトークン交換リクエスト」「ユーザー情報の取得」「データの暗号化」「セッション（JWT）の発行」という多くの「手順」を知りすぎています。
- **課題**: 認証プロバイダー（現在は Google のみ）が増えたり、ログイン時の追加処理が必要になった際、ハンドラー側の修正が肥大化する懸念があります。
- **推奨**: これらを `GoogleAuthService` や `SessionService` にカプセル化し、ハンドラーは「Service を呼んで、結果を Cookie にセットしてリダイレクトする」だけにするのが理想です。

### B. 同期ライフサイクル（PoC Handler / SyncOrchestrator）
`parsePendingHandler` 内で、`sync_runs`（実行ログ）の作成や終了時の更新を手動で行っています。
- **課題**: 「同期の実行そのもの」のログ管理は、ビジネスルールの中心部分です。
- **推奨**: `SyncOrchestrator` の中であらゆる「実行記録」の管理まで完結させるべきです。ハンドラーは「同期実行を開始せよ」という命令を送るだけに留めるべきです。

---

## 3. 推奨される今後のステップ（さらなる進化）

1.  **Service 層の拡充**:
    - `AuthService`: 認証〜ログイン処理の抽象化。
    - `SyncOrchestrator` の高度化: ライフサイクル管理（開始・終了・エラー記録）の内包。
2.  **Repository の Lazy Load**:
    - 現状全リポジトリを一括で `new` していますが、将来的に増えた場合はプロパティアクセス時に初めて `new` する「遅延評価」に切り替えること（内部実装のみの変更で対応可能）。
3.  **ミドルウェアの整理**:
    - 特定のルーティング（例：認証不要な画面）でのリポジトリ注入の要否など、プロジェクトの規模拡大に合わせた「ルーティング単位でのミドルウェア適用」の検討。

---

**結論**: 現時点のリファクタリングにより、プロジェクトの基盤は「非常に美しく、拡張可能な状態」になりました。将来的に機能が複雑化する際、上記の「Service 層へのさらなる移譲」を行う準備はすでに整っています。
