# 同期処理の実装レポートとトラブルシューティング (2026-03-10)

本ドキュメントは、Gym Booking Tracker のメール同期ロジック実装において発生した技術的課題とその解決策をまとめたものです。

## 🛠 トラブルシューティングと解決策

### 1. D1 データベースのスキーマ競合
- **問題**: マイグレーション適用時に `D1_ERROR: table sync_runs already exists` が発生し、ローカルDBの状態が不整合になった。
- **原因**: 開発中のマイグレーション履歴の管理ミス、または中断された実行により D1 が適用済みの変更を正しく認識できなかった。
- **解決策**: 
    - マイグレーションSQLに `DROP TABLE IF EXISTS` を追加。
    - `npx wrangler d1 execute gym-booking-db --local --command "..."` を使用して、手動でクリーンな状態にリセットしてから再実行。
- **教訓**: 開発環境ではマイグレーションのべき等性を確保し、状態が壊れた時のリセットコマンドを整備しておく。

### 2. エラーハンドリングにおける変数のスコープ
- **問題**: パース失敗時にログを記録しようとすると、プログラムがクラッシュした。
- **原因**: メール本文（`detail`）が `try` ブロック内で宣言されており、`catch` ブロックから参照できなかった。
- **解決策**: 宣言をループのスコープ（`let detail: any = null;`）に移動し、失敗時でも本文をログ（`error_detail`）に含められるように修正。

### 3. 日本語の文字化け (UTF-8)
- **問題**: 取得したメール本文の日本語が「化けて」読み取れなかった。
- **原因**: JavaScript 標準の `atob()` は、マルチバイトの UTF-8 シーケンスを考慮せずにデコードするため。
- **解決策**: `TextDecoder` を使用した堅牢なデコード関数を実装：
  ```typescript
  const decodeBase64 = (base64Url: string) => {
      const base64 = base64Url.replace(/-/g, '+').replace(/_/g, '/');
      const binString = atob(base64);
      const bytes = Uint8Array.from(binString, (m) => m.codePointAt(0)!);
      return new TextDecoder().decode(bytes);
  };
  ```
- **成果**: 日本語のメール本文を正しく取得、解析できるようになった。

### 4. パーサーの正規表現の堅牢性向上
- **問題**: 札幌市のメールであるにもかかわらず、100%パースに失敗していた。
- **原因**: `/(\d{2})月/` や `/(\d{2}:\d{2})/` という正規表現が厳密すぎて、`4月` や `9:00` といった1桁の数値にマッチしなかった。
- **解決策**: 
    - 数値部分を `\d{1,2}` に更新。
    - DB保存時にフォーマットが崩れないよう、`.padStart()` を用いて `2025-04-19 09:00` のようにゼロ埋め（正規化）を行うロジックを追加。
- **成果**: 有効な予約メールの成功率がほぼ100%になった。

### 5. 予約以外のメールの自動除外
- **問題**: 同じ件名/送信者で送られてくる「入力確認コード」などのメールも取得されてしまう。
- **理由**: Gmail の検索クエリが広範囲なため。
- **解決策**: パーサーが `null` を返したものを「想定内のパースエラー」としてログに記録し、同期処理自体は中断せず次の予約メールに進むように設計。
