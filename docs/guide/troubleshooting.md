# トラブルシューティング・開発ログ

開発中に発生した課題、エラー、およびその解決策を記録します。

## 2026-03-09: プロジェクト初期化と Gmail API PoC

### 1. プロジェクト初期化によるファイルの衝突
- **事象**: `README.md` や `wrangler.jsonc` が Hono の初期化コマンド (`npm create hono`) によって上書きされた。
- **原因**: カレントディレクトリに対してスクリプトを実行したため、既存ファイルがデフォルト値で上書きされた。
- **解決策**: 事前に作成していた要件定義の内容を、初期化後の `README.md` にマージして復元した。
- **教訓**: フレームワークの初期化コマンドを実行する際は、大切なファイルを一時避難させるか、Gitのコミットを活用して復元できるようにしておくこと。

### 2. Cloudflare Workers 環境での型エラー
- **事象**: `fetch` や `URL` などの組み込み関数が TypeScript でエラーになった。
- **原因**: 標準の DOM ライブラリではなく、Workers 特有の型定義が必要だった。
- **解決策**: `@cloudflare/workers-types` をインストールし、`tsconfig.json` の `compilerOptions.types` に追加した。

### 3. Gmail API: `invalid_grant` エラー (リフレッシュトークンの不備)
- **事象**: `Failed to refresh access token` エラーが発生し、Gmail API からデータが取得できなかった。
- **原因**: `.dev.vars` に設定した `GOOGLE_REFRESH_TOKEN` の末尾に、不要なドット (`.`) が含まれていた。
- **解決策**: ドットを削除して保存し、Wrangler を再起動（または自動検知）させた。
- **教訓**: トークンやシークレットをコピー＆ペーストする際は、末尾の記号や空白に細心の注意を払うこと。ログに `response.status` と `response.text()` を詳細に出力するようにしたことで原因特定が速まった。

### 4. 設定ファイルの Git 管理ポリシー
- **事象**: `wrangler.jsonc` を Git に含めて良いか迷った。
- **機密情報**: `npx wrangler secret put` で設定した値は Cloudflare サーバー側に、`.dev.vars` はローカルに閉じているため、`wrangler.jsonc` 自体には機密情報は含まれない。
- **結論**: インフラ構成（Binds 等）を共有するため、`wrangler.jsonc` は Git 管理対象とする。生のシークレットが記載された `memo.txt` 等は即座に削除する。

## 2026-03-10: データベース設計と永続化の導入

### 1. D1 のローカル開発時における接続先選択
- **事象**: `npx wrangler d1 create` や `wrangler dev` 起動時、"connect to the remote resource?" と聞かれる。
- **背景**: 「Yes」にするとクラウド上の本番DBを読み書きし、「No」にするとローカルの `.wrangler/state` 内の SQLite ファイルを読み書きする。
- **解決策**: 基本的に **No** を選択。本番データを汚染せず、高速に動作するため。
- **教訓**: ローカルでマイグレーションを実行する際も `--local` フラグを忘れないこと。

### 2. SQLite におけるデータ型の特性 (TEXT型)
- **事象**: `VARCHAR` や `DATE` 型を使わず、`TEXT` 型を多用する設計とした。
- **理由**: SQLite には厳密な長さ制限付きの `VARCHAR` は存在せず、内部的に `TEXT` として統一的に処理される。長さに関わらず、実際に格納された分のみストレージを消費するため、`TEXT` が最もシンプルかつ標準的な選択となる。
- **教訓**: 型による制限ではなく、Hono 等のアプリケーション層のバリデーション、または DB の `CHECK` 制約でデータ品質を担保する。

### 3. メールの解析とデータ品質
- **事象**: `2025年11月13日(木)18:15` のような日本語の日時をそのまま保存すると検索が困難になる。
- **解決策**: 解析（パース）の段階で ISO8601 形式 (`2025-11-13 18:15`) に正規化して、DB の `event_date` カラムに保存する。表示の際に必要に応じてフロントエンド側で日本語形式に戻す。

## 2026-03-31: アーキテクチャの近代化 (Factory Function & Biome)

### 1. クラスベースから Factory 関数形式への移行
- **事象**: クラスの静的メソッドのみを利用した場合の Biome 警告 (`noStaticOnlyClass`) や、`this` のバインドミスへの懸念。
- **原因**: 近代的な TypeScript 開発では、クラスよりもクロージャ（Factory 関数）を用いた関数型アプローチが好まれる傾向にある。
- **解決策**: すべてのサービス（Gmail, Sync, Auth, GoogleAuth）を Factory 関数形式にリファクタリング。`interface` を介した依存注入（DI）を徹底した。
- **教訓**: 初期段階で柔軟なアーキテクチャを採用しておくことで、テスタビリティ（テストのしやすさ）が飛躍的に向上する。

### 2. 開発ツールの Biome への統合
- **事象**: ESLint / Prettier の設定が複雑化し、動作が重くなっていた。
- **解決策**: すべてを [Biome](https://biomejs.dev/) に統合。`npm run check:all` スクリプトを新设し、型チェックと静的解析を高速に一括実行できる環境を構築した。
- **教訓**: 高速なリンター/フォーマッターの導入は、開発者体験 (DX) を直接的に改善し、コードの品質維持を容易にする。

### 3. 厳格な型 Promotion による安全性確保
- **事象**: Handler 内で `c.get('user')` 等を取得する際、型が `any` にになりがちだった。
- **解決策**: Hono のミドルウェアで `Context` の型の Promotion（昇格）を行い、ハンドラーに到達した時点ですべてのサービスやユーザー情報に型が正しくついている状態を実現した。
