## 1. 概要と共通設計

### 1.1 設計思想
本プロジェクトの API は、フロントエンドとバックエンドの型安全性を担保するため、[Hono Zod OpenAPI](https://hono.dev/examples/zod-openapi) を採用しています。全ての型定義（スキーマ）は実装コードと直接同期されており、技術的な詳細仕様については実装から自動生成される **Swagger UI** が正解（Single Source of Truth）となります。

本ドキュメントでは、各機能の目的や裏側の動作、ユーザー体験の設計といった「動作仕様」を日本語で解説します。

### 1.2 ドキュメントの役割
- **`/swagger`**: 最新の技術仕様（エンドポイント、パラメータ、レスポンスの型定義）。
- **`api-design.md`（本紙）**: 各 API 群の役割、依存関係、非同期処理などの論理的な挙動。

### 1.3 共通レスポンス構造
API は、クライアントが処理の成否を直感的に判断できるよう、以下の共通構造を基本としています。

- **`success`**: 処理の成否を示すブール値。
- **`message`**: 人間が読める形式の実行結果メッセージ。
- **`data`**: 成功時の返却データ。
- **`error`**: 失敗時のエラー情報。`code`（機械判別用）と `message`（デバッグ用）を含みます。

## 2. 認証 (Authentication)

Google OAuth2.0 を利用した認証フローを採用しています。

### 2.1 認証プロセス
1.  **認可の開始**: ユーザーが `/auth/google` にアクセスすると、Google のログイン画面へリダイレクトされます。
2.  **コールバック**: ログイン完了後、Google から `/auth/google/callback` へ認可コードが送られます。
3.  **セッション確立**: バックエンドでトークン交換とユーザー情報の検証を行い、`auth_token` クッキーを発行します。
4.  **画面遷移**: 認証成功後はクリーンな URL である `/auth/success` を経由してダッシュボードへと誘導されます。

### 2.2 セッション管理
- 認証は HttpOnly なクッキーベースで行われます。
- `/auth/logout` にアクセスすることで、クッキーが削除されセッションが終了します。

## 3. 同期システム (Mail Synchronization)

Gmail からジムの予約メールを自動取得し、解析・保存する一連の流れを管理します。

### 3.1 同期のフェーズ
同期処理は、以下の2つのフェーズに分かれています。
- **Ingest (取得)**: Gmail API を使用し、特定の検索クエリ（ジムの予約メール等）に一致する最新のメールを DB (`raw_emails` テーブル) に取り込みます。
- **Parse (解析)**: DB に保存された未処理のメール本文を順次読み込み、施設名、日時などの情報を抽出して「予約データ」として保存します。

### 3.2 フル同期の挙動 (`POST /sync/`)
- このエンドポイントは**非同期**で動作します。
- リクエストを受けると、Cloudflare Workers の `waitUntil` 処理により、レスポンス返却後にバックグラウンドで Ingest と Parse を連続して実行します。
- クライアントには即座に `runId` が返却され、進捗は `/sync/:runId/status` で確認できます。

### 3.3 データのリセット (`DELETE /sync/data`)
- 開発や再テストを容易にするため、現在のユーザーに関連する全ての予約、メール、同期ログを物理削除します。

## 4. 予約管理 (Booking Management)

解析済みの予約情報を管理・取得するための機能です。

### 4.1 データの取得
- `/bookings` では、最新順にソートされた解析済みデータ一覧を返却します。
- 同期のタイミングによっては、ステータスが「自動更新待ち」となっている場合があります。

### 4.2 今後の予定 (Planned)
- **詳細表示 (`GET /bookings/:id`)**: 特定の予約に関連する抽出元のメール情報や、より詳細なメタデータを確認可能にします。
- **ステータス更新 (`PATCH /bookings/:id`)**: 手動でのキャンセルマークや、メモの追記を可能にする計画です。

---

## 5. デバッグ機能 (Debug)

開発効率を向上させるための補助メニューです。

### 5.1 データの部分削除 (`POST /debug/clear-latest`)
- 全削除ではなく「最新の N 件分だけを解析前の状態に戻してやり直す」といった作業を可能にします。
- 解析エンジンのプロンプト調整などの反復開発に利用されます。
