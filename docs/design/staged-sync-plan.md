# 同期アーキテクチャ刷新：設計書 (2026-03-12)

Gmailからのメール同期処理を「生のデータの取り込み（インジェスト）」と「データ解析（パース）」の2段階に分離し、大量データの処理と自動実行（Cron）に対応させます。

> [!IMPORTANT]
> **進行方針**
> 本設計書の内容をユーザー（k-sugawara様）が確認・レビューし、**承認をいただいた後に実装作業を開始します。**

## 1. 目的
- **堅牢性の向上**: 解析エラーが発生しても生データがDBに残るため、ロジック修正後の再実行を容易にする。
- **データ取得の完全性**: 500件を超える大量のメールも、ページネーションを用いて漏れなく取得する。
- **運用の自動化**: HTTPリクエスト制限を受けない Cron (Scheduled) 実行を可能にする。

## 2. データベース設計の変更 (D1)
生のメールを一時保存するテーブルを追加します。

### `raw_emails` テーブル [新規]
- `id`: メールID (Gmail Message ID) - PK
- `thread_id`: スレッドID
- `subject`: メールの件名（タイトル）
- `snippet`: スニペット（概要文）
- `body`: デコード済みの本文全文
- `fetched_at`: 取り込み日時 (Unix時間)
- `parse_status`: 解析状態 (`pending`, `success`, `fail`)

## 3. 実装の設計

### Gmail連携 (`GmailService`)
- `listMessages` メソッドを拡張し、`nextPageToken` を処理できるようにして、クエリに一致するすべてのメッセージをループで取得可能にします。
- `getMessage` メソッドで、ヘッダーから `Subject`（タイトル）を抽出するようにします。

### 同期ロジック (`SyncOrchestrator`)
- **`ingest()`**: Gmailから未登録のメールをすべて取得し、`raw_emails` テーブルに「未処理」として保存します。
- **`processPending()`**: DB内の未処理データを読み込み、パースを実行。成功・失敗に応じて `parse_status` を更新し、`bookings` テーブルに反映します。
- **`sync()`**: 上記のインジェストとパースを順番に呼び出すメイン処理。

### 自動実行 (Cron Job / Scheduled)
- `wrangler.jsonc` にスケジュール（例：1時間ごと）を定義。
- `src/index.ts` に `scheduled` イベントハンドラを追加し、`SyncOrchestrator.sync()` を実行するようにします。

## 4. 検証予定
- インジェスト実行後、`raw_emails` に全件正しく保存されるか。
- パース実行後、`bookings` が更新され、解析状態が `success` になるか。
- すでにDBにあるメールが、二重に処理されないか（冪等性の確認）。
