# Development Rules

## 1. Project Goal

公共施設予約システムの当選・申し込み情報を自動的に集約し、アカウント横断で一元管理することで、ユーザーの確認負荷を完全に排除すること。

## 2. Core Principles

AIがコード生成やリファクタリングを提案する際は、常に以下の判断基準を通すこと。

- Privacy First:
    - 施設名、特定の地名、個人の特定につながる情報はコード内にハードコードせず、必ず環境変数または設定ファイル経由で読み込むこと。
- Modularity & Decoupling:
    - メールのパースロジック(解析)と、データ保存・ビジネスロジックを分離すること。これにより、メールのフォーマット変更に強く、テスタブルな構造を維持する。
- Event-Driven Resilience:
    - 常駐サーバーを立てない。Cloudflare Workers (Cron/Webhook)で完結させ、失敗しても再試行が容易な冪等性を意識した設計にする。
- Type Safety:
    - TypeScriptの型定義を最大限活用する。全てのデータは明確なインターフェースを定義してから実装に着手すること。

## 3. Workflow Priorities

全ての実装は以下の順序で進行する。

1. Define: まずインターフェース(型)を定義する
2. Implement: ロジックを実装する
3. Validate: `antigravity`のテスト生成機能を活用し、異常系を含むテストを書く
4. Document: 要件に変更がある場合は実装後にREADME.md(要件定義書)を更新し、ドキュメントを最新に保つ

## 4. Constraint for AI
- 外部APIへのアクセスや複雑なロジックを追加する際は、必ず事前にこの`rules.md`の目的と合致しているかを確認すること。
- コードが「なぜこの実装になったのか」という意図(Why)をコメントに残すこと。
- 出力は日本語とすること。
- APIエンドポイントを追加・修正した場合は、必ず `docs/openapi.yaml` を確認し、必要であれば最新の仕様を反映すること。
